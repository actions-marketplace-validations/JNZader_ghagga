## Fortalezas Detectadas

- TypeScript strict en todo el monorepo, uso mínimo de `any` (solo en tests para mocks)
- Criptografía excelente: AES-256-GCM con IV random por operación, `timingSafeEqual()` para HMAC
- 1,787+ tests en 126 archivos, con mutation testing (Stryker) configurado en core/action/server
- Graceful degradation en todos los niveles — si memory, static analysis o un provider falla, el sistema continúa
- Privacy stripping antes de persistir en memoria (regex para API keys, tokens, PEM, JWT)
- Arquitectura limpia: separación core/db/apps, sin dependencias circulares, provider factory con Vercel AI SDK
- Docker multi-stage con imágenes slim y dev deps purgadas
- OIDC provenance en npm publish
- Sin eval(), sin SQL injection (Drizzle ORM), sin command injection
- Exhaustive type checking con `never` en switches

## Hallazgos y Propuestas de Mejora

### CRÍTICO

#### 1. CORS permite todos los orígenes

**Archivo**: `apps/server/src/index.ts` ~línea 74
**Problema**: `app.use('*', cors())` sin restricción de origen. Cualquier dominio puede hacer requests a la API, que maneja API keys encriptadas y tokens de GitHub.

**Cambio propuesto**:
```typescript
app.use('*', cors({
  origin: [
    'https://jnzader.github.io',
    'https://ghagga.onrender.com',
    ...(process.env.NODE_ENV !== 'production' ? ['http://localhost:3000', 'http://localhost:5173'] : []),
  ],
  credentials: true,
}));
```

**Esfuerzo**: ~15 minutos
**Impacto**: Alto — cierra vector de ataque cross-origin

#### 2. Sin Rate Limiting en endpoints de API

**Archivos afectados**: `apps/server/src/routes/api.ts`, `apps/server/src/routes/oauth.ts`
**Problema**: Todos los endpoints `/api/*` y `/oauth/*` están sin rate limiting. Vulnerables a enumeración, abuso, y DDoS a nivel de aplicación.

**Endpoints expuestos**:
- `GET /api/reviews` — query por `repo`
- `GET /api/stats` — query por `repo`
- `GET /api/repositories` — lista todas
- `GET /api/installations` — lista todas
- `DELETE /api/memory/*` — eliminación masiva
- `GET /oauth/login` — inicio de OAuth
- `GET /oauth/callback` — intercambio de token

**Cambio propuesto**: Implementar rate limiting con middleware de Hono.

```typescript
import { rateLimiter } from 'hono-rate-limiter';

// Limitar endpoints de API: 100 req/min por IP
const apiLimiter = rateLimiter({
  windowMs: 60 * 1000,
  limit: 100,
  keyGenerator: (c) => c.req.header('x-forwarded-for') || 'unknown',
});

// Limitar OAuth: 10 req/min por IP
const oauthLimiter = rateLimiter({
  windowMs: 60 * 1000,
  limit: 10,
  keyGenerator: (c) => c.req.header('x-forwarded-for') || 'unknown',
});
```

**Esfuerzo**: ~1 hora (incluye dependencia + tests)
**Impacto**: Alto — previene abuso de API

#### 3. PEM file en historial de Git

**Commit**: `c743ab9`
**Problema**: La private key original fue commiteada y luego removida. Aunque fue reemplazada por otra key, el archivo sigue accesible en el historial de git.

**Cambio propuesto**: Si el repo es público, limpiar el historial:

```bash
# Opción 1: git filter-repo (recomendado)
git filter-repo --path ghagga-review.2026-03-02.private-key.pem --invert-paths

# Opción 2: BFG Repo Cleaner
bfg --delete-files 'ghagga-review.2026-03-02.private-key.pem'
git reflog expire --expire=now --all && git gc --prune=now --aggressive
```

**Nota**: Requiere force push. Coordinar con colaboradores.
**Esfuerzo**: ~30 minutos
**Impacto**: Medio (key ya fue rotada, pero es higiene de seguridad)

### ALTO

#### 4. Sin ESLint ni Prettier/Biome

**Problema**: No existe NINGUNA configuración de linting o formatting en el proyecto. No hay enforcement automático de estilo ni detección de errores estáticos. La calidad depende 100% de reviews manuales.

**Cambio propuesto**: Configurar Biome (más rápido que ESLint + Prettier combinados):

```bash
pnpm add -Dw @biomejs/biome
```

Crear `biome.json` en la raíz:
```json
{
  "$schema": "https://biomejs.dev/schemas/2.0.0/schema.json",
  "organizeImports": { "enabled": true },
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true,
      "correctness": {
        "noUnusedVariables": "warn",
        "noUnusedImports": "warn"
      },
      "suspicious": {
        "noExplicitAny": "warn"
      }
    }
  },
  "formatter": {
    "indentStyle": "space",
    "indentWidth": 2,
    "lineWidth": 100
  },
  "files": {
    "ignore": ["node_modules", "dist", "coverage", ".turbo"]
  }
}
```

Agregar scripts al root `package.json`:
```json
{
  "lint": "biome check .",
  "lint:fix": "biome check --write ."
}
```

**Esfuerzo**: ~2 horas (setup + fix inicial de warnings)
**Impacto**: Alto — previene drift de calidad a largo plazo

#### 5. Sin security scanning en CI

**Archivo**: `.github/workflows/ci.yml`
**Problema**: El CI hace typecheck + build + test, pero NO corre análisis de seguridad de dependencias ni imágenes Docker.

**Cambio propuesto**: Agregar job de seguridad al CI:

```yaml
security:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: pnpm/action-setup@v4
    - uses: actions/setup-node@v4
      with:
        node-version: 22
        cache: 'pnpm'
    - run: pnpm install --frozen-lockfile
    - name: Audit dependencies
      run: pnpm audit --prod
    - name: Run Trivy (filesystem scan)
      uses: aquasecurity/trivy-action@master
      with:
        scan-type: 'fs'
        severity: 'CRITICAL,HIGH'
```

**Esfuerzo**: ~1 hora
**Impacto**: Alto — detecta vulnerabilidades conocidas en dependencias

#### 6. Coverage no se reporta ni trackea en CI

**Archivo**: `.github/workflows/ci.yml`
**Problema**: Los tests corren pero los resultados de coverage no se suben. No hay thresholds — se puede mergear un PR que baje la cobertura sin aviso.

**Cambio propuesto**: Agregar upload de coverage al job de tests:

```yaml
- name: Run tests with coverage
  run: pnpm exec turbo test:coverage
  env:
    ENCRYPTION_KEY: '0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef'

- name: Upload coverage
  uses: codecov/codecov-action@v4
  with:
    token: ${{ secrets.CODECOV_TOKEN }}
    files: packages/core/coverage/coverage-summary.json,apps/action/coverage/coverage-summary.json
    fail_ci_if_error: false
```

**Esfuerzo**: ~30 minutos
**Impacto**: Medio-Alto — visibilidad de tendencias de cobertura

#### 7. `console.error` en lugar de logger estructurado

**Archivo**: `apps/server/src/routes/oauth.ts` líneas 207, 221

**Problema**:
```typescript
console.error('STATE_SECRET is not configured');
console.error('GITHUB_CLIENT_SECRET is not configured');
```
El server tiene Pino configurado con redaction automática de secrets. Usar `console.error` bypasea toda esa infraestructura.

**Cambio propuesto**: Reemplazar con logger:
```typescript
logger.error('STATE_SECRET is not configured');
logger.error('GITHUB_CLIENT_SECRET is not configured');
```

**Esfuerzo**: ~5 minutos
**Impacto**: Bajo (no es un riesgo de seguridad en sí, pero es inconsistencia)

#### 8. OAuth Client ID hardcodeado en 2 lugares

**Archivos**:
- `apps/server/src/routes/oauth.ts` línea 23: `'Ov23liyYpSgDqOLUFa5k'`
- `apps/dashboard/src/lib/oauth.ts` línea 16: `'Ov23liyYpSgDqOLUFa5k'`

**Problema**: Viola DRY y dificulta rotación de credenciales. Técnicamente los Client IDs son públicos por spec de OAuth, pero hardcodearlos en múltiples lugares es mala práctica.

**Cambio propuesto**:
- Server: mover a `process.env.GITHUB_CLIENT_ID`
- Dashboard: mover a `import.meta.env.VITE_GITHUB_CLIENT_ID`

**Esfuerzo**: ~15 minutos
**Impacto**: Medio — facilita rotación y multi-environment

### MEDIO

#### 9. Sin security headers

**Archivo**: `apps/server/src/index.ts`
**Problema**: No hay middleware de security headers. Falta CSP, X-Frame-Options, X-Content-Type-Options, HSTS.

**Cambio propuesto**: Agregar middleware de headers:

```typescript
import { secureHeaders } from 'hono/secure-headers';

app.use('*', secureHeaders({
  contentSecurityPolicy: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'"],
  },
  xFrameOptions: 'DENY',
  xContentTypeOptions: 'nosniff',
  strictTransportSecurity: 'max-age=31536000; includeSubDomains',
}));
```

**Esfuerzo**: ~30 minutos
**Impacto**: Medio — hardening de la aplicación web

---

#### 10. Sin request size limits

**Archivo**: `apps/server/src/index.ts`
**Problema**: No hay validación del tamaño del body en endpoints. Un payload gigante podría causar OOM.

**Cambio propuesto**: Agregar middleware de body limit:

```typescript
import { bodyLimit } from 'hono/body-limit';

app.use('*', bodyLimit({ maxSize: 1024 * 1024 })); // 1MB
```

**Esfuerzo**: ~10 minutos
**Impacto**: Medio — previene OOM por payloads maliciosos

#### 11. Mutation tests no corren en CI

**Archivos**: `packages/core/stryker.config.json`, `apps/action/stryker.config.json`, `apps/server/stryker.config.mjs`
**Problema**: Stryker está configurado con thresholds (80/60) pero solo se ejecuta manualmente.

**Cambio propuesto**: Agregar como job opcional en CI (solo en main, no en PRs por tiempo):

```yaml
mutation:
  runs-on: ubuntu-latest
  if: github.ref == 'refs/heads/main'
  needs: test
  steps:
    - uses: actions/checkout@v4
    - uses: pnpm/action-setup@v4
    - uses: actions/setup-node@v4
      with:
        node-version: 22
        cache: 'pnpm'
    - run: pnpm install --frozen-lockfile
    - run: pnpm exec turbo build
    - run: cd packages/core && pnpm test:mutation
```

**Esfuerzo**: ~30 minutos
**Impacto**: Medio — detecta tests que no matan mutantes

const reviewsByDay = await db
