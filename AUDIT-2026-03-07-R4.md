## Hallazgos y Propuestas de Mejora

### CRÍTICO — Production Blockers

#### 1. Sin HTTP timeouts en llamadas a GitHub API

**Archivos**: `apps/server/src/github/client.ts`, `apps/server/src/github/runner.ts`
**Problema**: Ninguna llamada `fetch()` tiene timeout. Si GitHub responde lento, el request cuelga indefinidamente, agotando el connection pool.

**Cambio propuesto**: Agregar `AbortSignal.timeout()` a todas las llamadas:

```typescript
const response = await fetch(url, {
  signal: AbortSignal.timeout(10_000), // 10 segundos
  headers: { Authorization: `Bearer ${token}`, ... },
});
```

**Esfuerzo**: 1 hora
**Impacto**: Crítico — previene resource exhaustion

---

#### 2. Validación de env vars al startup (fail-fast)

**Archivo**: `apps/server/src/index.ts`
**Problema**: El server inicia correctamente aunque falten `GITHUB_APP_ID`, `GITHUB_PRIVATE_KEY`, `GITHUB_WEBHOOK_SECRET`, etc. Recién falla cuando llega el primer webhook (minutos después), sin mensaje claro.

**Cambio propuesto**:

```typescript
function validateEnvironment() {
  const required = [
    'DATABASE_URL',
    'GITHUB_APP_ID',
    'GITHUB_PRIVATE_KEY',
    'GITHUB_WEBHOOK_SECRET',
    'ENCRYPTION_KEY',
  ];
  const missing = required.filter(key => !process.env[key]);
  if (missing.length > 0) {
    logger.error({ missing }, `Missing required env vars: ${missing.join(', ')}`);
    process.exit(1);
  }
}

// Llamar ANTES de inicializar el server
validateEnvironment();
const db = createDatabaseFromEnv();
```

**Esfuerzo**: 30 minutos
**Impacto**: Crítico — fail-fast en lugar de fail-late

---

#### 3. FK cascade delete incompleto en schema

**Archivo**: `packages/db/src/schema.ts`
**Problema**: La migración 0005 arregló cascade delete en `memory_observations`, pero otros FKs siguen con `ON DELETE no action`:
- `github_user_mappings.installation_id` → `installations.id`
- `repositories.installation_id` → `installations.id`
- `reviews.repository_id` → `repositories.id`

Cuando se desinstala una GitHub App, los repositories y reviews quedan huérfanos.

**Cambio propuesto**: Nueva migración que arregle los FKs restantes:

```sql
ALTER TABLE repositories DROP CONSTRAINT IF EXISTS repositories_installation_id_fkey;
ALTER TABLE repositories ADD CONSTRAINT repositories_installation_id_fkey
  FOREIGN KEY (installation_id) REFERENCES installations(id) ON DELETE CASCADE;

ALTER TABLE reviews DROP CONSTRAINT IF EXISTS reviews_repository_id_fkey;
ALTER TABLE reviews ADD CONSTRAINT reviews_repository_id_fkey
  FOREIGN KEY (repository_id) REFERENCES repositories(id) ON DELETE CASCADE;

ALTER TABLE github_user_mappings DROP CONSTRAINT IF EXISTS github_user_mappings_installation_id_fkey;
ALTER TABLE github_user_mappings ADD CONSTRAINT github_user_mappings_installation_id_fkey
  FOREIGN KEY (installation_id) REFERENCES installations(id) ON DELETE CASCADE;
```

**Esfuerzo**: 1 hora (migración + test)
**Impacto**: Crítico — previene datos huérfanos y potenciales FK violations

### ALTO

#### 4. Sin correlation IDs para tracing de reviews

**Problema**: No hay forma de trazar un review completo desde webhook → Inngest → pipeline → comment. Si un cliente reporta "mi review falló en PR 123", hay que matchear timestamps manualmente en los logs.

**Cambio propuesto**: Generar `reviewId` en el webhook y propagarlo:

```typescript
// webhook.ts
const reviewId = crypto.randomUUID().slice(0, 8);
await inngest.send({
  name: 'ghagga/review.requested',
  data: { reviewId, installationId, repoFullName, ... },
});

// inngest/review.ts
const { reviewId } = event.data;
const logger = rootLogger.child({ reviewId });
// Todos los logs de esta review incluyen reviewId
```

**Esfuerzo**: 3-4 horas
**Impacto**: Alto — esencial para debugging en producción

#### 5. Token cache: cleanup solo por threshold, no periódico

**Archivo**: `apps/server/src/middleware/auth.ts`
**Problema**: El cleanup de tokens expirados solo corre cuando el cache supera 1000 entries. En deployments de bajo tráfico, tokens expirados se acumulan indefinidamente.

**Cambio propuesto**: Agregar cleanup periódico:

```typescript
setInterval(() => {
  const now = Date.now();
  for (const [key, value] of tokenCache) {
    if (value.expiresAt <= now) tokenCache.delete(key);
  }
}, 5 * 60 * 1000); // Cada 5 minutos
```

**Esfuerzo**: 15 minutos
**Impacto**: Medio-Alto — previene memory leak lento

#### 6. Keepalive fetch sin timeout

**Archivo**: `apps/server/src/index.ts` ~líneas 216-231
**Problema**: El keepalive a Render no tiene timeout. Si el request cuelga, bloquea un tick del event loop.

**Cambio propuesto**:

```typescript
await fetch(url, { signal: AbortSignal.timeout(5000) });
```

**Esfuerzo**: 5 minutos
**Impacto**: Medio

#### 7. GitHub API pagination trunca PRs con >100 archivos

**Archivo**: `apps/server/src/github/client.ts`
**Problema**: `per_page=100` en `getPRFileList()` y `getPRCommitMessages()`. PRs de monorepos grandes con >100 archivos se truncan silenciosamente.

**Cambio propuesto**: Paginar o documentar la limitación. Para archivos, implementar paginación:

```typescript
async function getPRFileList(owner, repo, prNumber, token) {
  const files: string[] = [];
  let page = 1;
  while (true) {
    const response = await fetch(`...?per_page=100&page=${page}`, ...);
    const data = await response.json();
    files.push(...data.map(f => f.filename));
    if (data.length < 100) break;
    page++;
  }
  return files;
}
```

**Esfuerzo**: 1 hora
**Impacto**: Medio-Alto — data truncation silenciosa en monorepos

#### 12. Migraciones no idempotentes

**Archivo**: `packages/db/src/migrate.ts`
**Problema**: Si una migración custom (ej: `0001_add_tsvector.sql`) se ejecuta dos veces, `CREATE TRIGGER` falla porque ya existe. Las migraciones no están wrapeadas en transacciones ni tienen guards `IF NOT EXISTS`.

**Esfuerzo**: 2 horas
**Impacto**: Medio — deployment reliability

---

#### 13. Dockerfile sin HEALTHCHECK

**Archivo**: `apps/server/Dockerfile`
**Problema**: No tiene instrucción `HEALTHCHECK`. Docker/Docker Compose no pueden verificar que el container está healthy.

**Cambio propuesto**:

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=10s --retries=3 \
  CMD ["node", "-e", "fetch('http://localhost:3000/health').then(r => process.exit(r.ok ? 0 : 1))"]
```

**Esfuerzo**: 15 minutos
**Impacto**: Medio — container orchestration

### BAJO

#### 14. Sin métricas estructuradas (reviews/day, duration, provider usage)

**Problema**: No hay counters ni histogramas. En producción no se puede responder: "¿Cuántas reviews corremos por día? ¿Cuánto tardan? ¿Qué provider se usa más?"

**Cambio propuesto**: Agregar structured metrics en los logs de Pino:

```typescript
logger.info({
  metrics: {
    durationMs: Date.now() - startTime,
    provider: primary.provider,
    status: result.status,
    findingsCount: result.findings.length,
  }
}, 'Review completed');
```

**Esfuerzo**: 2-3 horas
**Impacto**: Bajo-Medio — observabilidad

---

#### 15. API response envelope inconsistente

**Problema**: La mayoría de endpoints responden `{ data: ... }`, pero `PUT /api/settings` responde `{ message: 'Settings updated' }` sin wrapper `data`.

**Esfuerzo**: 15 minutos
**Impacto**: Bajo

---

#### 16. Error IDs en respuestas de API para soporte

**Problema**: Cuando una API devuelve 500, el usuario no tiene referencia para reportar al soporte. Si se incluye un `errorId` en la respuesta y en el log, el soporte puede buscar directamente.

```typescript
const errorId = crypto.randomUUID().slice(0, 8);
logger.error({ err, errorId }, 'Failed to fetch settings');
return c.json({ error: 'FETCH_FAILED', message: '...', errorId }, 500);
```

**Esfuerzo**: 1 hora
**Impacto**: Bajo — mejora UX de soporte

## Verificaciones Positivas
