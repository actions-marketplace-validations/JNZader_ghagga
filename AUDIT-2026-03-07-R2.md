## Hallazgos y Propuestas de Mejora

### CRÍTICO

#### 1. Stack traces expuestos en modo desarrollo

**Archivo**: `apps/server/src/index.ts` ~líneas 37-51
**Problema**: El error handler global incluye `err.message` y `err.stack` cuando `NODE_ENV !== 'production'`. Si un deployment de dev está expuesto a internet, se filtran rutas internas, esquema de DB, y detalles de librerías.

```typescript
// ACTUAL
return c.json({
  error: 'Internal server error',
  ...(process.env.NODE_ENV !== 'production' && {
    detail: err.message,
    stack: err.stack  // ← EXPUESTO
  }),
}, 500);
```

**Cambio propuesto**: Loguear el error completo con el logger estructurado, devolver solo el mensaje genérico al cliente:

```typescript
app.onError((err, c) => {
  logger.error({ err, method: c.req.method, path: new URL(c.req.url).pathname },
    'Unhandled error');
  return c.json({ error: 'Internal server error' }, 500);
});
```

**Esfuerzo**: 15 minutos
**Impacto**: Crítico — previene information disclosure

#### 2. Endpoint de memory sessions sin autorización cross-installation

**Archivo**: `apps/server/src/routes/api.ts` ~líneas 640-653
**Problema**: `GET /api/memory/sessions/:id/observations` solo requiere autenticación, pero NO verifica que la sesión pertenezca a una instalación del usuario. Un usuario autenticado puede leer observaciones de CUALQUIER sesión (incluidas las de otros equipos).

**Cambio propuesto**: Verificar ownership de la sesión antes de devolver datos:

```typescript
router.get('/api/memory/sessions/:id/observations', async (c) => {
  const user = c.get('user') as AuthUser;
  const sessionId = parseInt(c.req.param('id'), 10);

  // Verificar que la sesión pertenece a una instalación del usuario
  const session = await getSessionById(db, sessionId);
  if (!session) return c.json({ error: 'Session not found' }, 404);

  const repo = await getRepoByFullName(db, session.project);
  if (!repo || !user.installationIds.includes(repo.installationId)) {
    return c.json({ error: 'Forbidden' }, 403);
  }

  const observations = await getObservationsBySession(db, sessionId);
  return c.json({ data: observations });
});
```

**Esfuerzo**: 30 minutos
**Impacto**: Crítico — previene acceso cross-team a datos de memoria

### ALTO

#### 3. N+1 queries en endpoints de API

**Archivo**: `apps/server/src/routes/api.ts` ~líneas 138-142, 157-166
**Problema**: Los endpoints de repositories e installations hacen queries secuenciales en un loop:

```typescript
for (const installationId of user.installationIds) {
  const repos = await getReposByInstallationId(db, installationId);
  allRepos.push(...repos);
}
```

Si un usuario tiene 10 instalaciones, son 10 queries secuenciales (500ms+ de latencia).

**Cambio propuesto**: Paralelizar con `Promise.all()` o crear query batch:

```typescript
// Opción 1: Paralelizar
const repoArrays = await Promise.all(
  user.installationIds.map(id => getReposByInstallationId(db, id))
);
const allRepos = repoArrays.flat();

// Opción 2: Query batch (mejor)
const allRepos = await getReposByInstallationIds(db, user.installationIds);
```

**Esfuerzo**: 1 hora
**Impacto**: Alto — 90% reducción de latencia para usuarios con múltiples instalaciones

#### 4. Auth middleware llama a GitHub API en CADA request

**Archivo**: `apps/server/src/middleware/auth.ts` ~líneas 62-102
**Problema**: Cada request a `/api/*` hace un `fetch('https://api.github.com/user')` para validar el token. Esto agrega 100-500ms de latencia a cada request y puede causar rate limiting de GitHub.

**Cambio propuesto**: Cache de tokens con TTL:

```typescript
const tokenCache = new Map<string, { user: GitHubUser; expiresAt: number }>();
const CACHE_TTL_MS = 5 * 60 * 1000; // 5 minutos

function getCachedUser(token: string): GitHubUser | null {
  const cached = tokenCache.get(token);
  if (cached && cached.expiresAt > Date.now()) return cached.user;
  tokenCache.delete(token);
  return null;
}
```

**Esfuerzo**: 1 hora
**Impacto**: Alto — 90% menos llamadas a GitHub API, latencia <5ms por request cacheado

#### 5. Sin Error Boundary en el dashboard React

**Archivo**: `apps/dashboard/src/App.tsx`
**Problema**: No hay ningún `<ErrorBoundary>` en la app. Si un componente de cualquier página crashea (error de render), toda la aplicación se cae sin posibilidad de recuperación.

**Cambio propuesto**: Crear componente ErrorBoundary y wrappear las rutas:

```tsx
// components/ErrorBoundary.tsx
class ErrorBoundary extends Component<Props, State> {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error };
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback error={this.state.error} onReset={() => this.setState({ hasError: false })} />;
    }
    return this.props.children;
  }
}
```

**Esfuerzo**: 1 hora
**Impacto**: Alto — la app se recupera de errores de render en lugar de crashear

#### 6. Archivo api.ts de 912 líneas con múltiples responsabilidades

**Archivo**: `apps/server/src/routes/api.ts`
**Problema**: Un solo archivo con 10+ endpoints que maneja autenticación, business logic, formateo de datos, error handling, y logging. Difícil de navegar, testear, y extender.

**Cambio propuesto**: Dividir por dominio:

```
routes/api/

  ├── repositories.ts   (GET /api/repositories)
  ├── installations.ts  (GET/PUT /api/installations, settings)
  ├── memory.ts         (GET/DELETE /api/memory/*)
  └── runner.ts         (POST /api/runner/create)
```

**Esfuerzo**: 2-3 horas
**Impacto**: Alto — mantenibilidad, testabilidad, y onboarding de contributors

#### 7. Sin contract testing entre dashboard y server

**Problema**: Los tipos de API están duplicados:
- Dashboard define tipos en `apps/dashboard/src/lib/types.ts`
- Server define tipos inline en los route handlers
- No hay paquete compartido de tipos

Si el server renombra un campo (ej: `aiReviewEnabled` → `aiEnabled`), los tests de ambos lados pasan pero la app se rompe en runtime.

**Cambio propuesto**: Crear `packages/types` con tipos compartidos, o usar esquemas Zod que validen en ambos extremos.

**Esfuerzo**: 3-4 horas
**Impacto**: Alto — previene breaking changes silenciosos entre frontend y backend

#### 8. Sin graceful shutdown

**Archivo**: `apps/server/src/index.ts`
**Problema**: El server no maneja SIGTERM. Cuando el container se detiene (redeploy en Render), los requests in-flight se cortan sin esperar. Reviews a mitad de guardado se pierden.

**Cambio propuesto**:

```typescript
const server = serve({ fetch: app.fetch, port }, () => { /* ... */ });

process.on('SIGTERM', () => {
  logger.info('Received SIGTERM, draining connections...');
  server.close(() => {
    logger.info('Server closed gracefully');
    process.exit(0);
  });
  // Force exit after 30s
  setTimeout(() => process.exit(1), 30_000);
});
```

**Esfuerzo**: 30 minutos
**Impacto**: Alto — previene pérdida de datos en redeploys

### MEDIO

#### 9. GitHub Actions no pinneados a SHAs

**Archivos**: `.github/workflows/ci.yml`, `publish.yml`, `deploy-pages.yml`
**Problema**: Las actions usan tags flotantes (`@v4`) en vez de commit SHAs. Un compromiso del repo de la action podría inyectar código.

**Cambio propuesto**: Pinear a SHAs específicos:

```yaml
# En vez de:
- uses: actions/checkout@v4
# Usar:
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2
```

**Esfuerzo**: 30 minutos
**Impacto**: Medio — supply chain hardening

#### 10. Docker base images no pinneadas a digest

**Archivos**: `apps/action/Dockerfile`, `apps/server/Dockerfile`
**Problema**: `FROM node:22-slim` usa tag flotante. Actualizaciones del base image pueden introducir breaking changes.

**Cambio propuesto**: Pinear a digest específico:

```dockerfile
FROM node:22-slim@sha256:<digest> AS builder
```

**Esfuerzo**: 30 minutos
**Impacto**: Medio — reproducibilidad de builds

#### 11. Sin circuit breaker para GitHub API

**Archivo**: `apps/server/src/github/client.ts`
**Problema**: Si GitHub está caído, cada webhook y cada request del dashboard bloquea esperando timeout. No hay fast-fail automático.

**Cambio propuesto**: Implementar circuit breaker (con `opossum` o similar):

```typescript
import CircuitBreaker from 'opossum';

const githubBreaker = new CircuitBreaker(githubApiCall, {
  timeout: 5000,
  errorThresholdPercentage: 50,
  resetTimeout: 30000,
});
```

**Esfuerzo**: 2 horas
**Impacto**: Medio — resiliencia cuando GitHub tiene problemas

#### 13. Lógica de provider chain view duplicada 3+ veces

**Archivo**: `apps/server/src/routes/api.ts` ~líneas 199-206, 384-391, 400-407
**Problema**: `maskApiKey()` y el mapeo de provider chain se repite en 3+ lugares. Viola DRY.

**Cambio propuesto**: Extraer helper:

```typescript
function buildProviderChainView(chain: DbProviderChainEntry[]) {
  return chain.map((entry) => ({
    provider: entry.provider,
    model: entry.model,
    hasApiKey: entry.encryptedApiKey != null,
    maskedApiKey: entry.encryptedApiKey ? maskApiKey(decrypt(entry.encryptedApiKey)) : undefined,
  }));
}
```

**Esfuerzo**: 30 minutos
**Impacto**: Medio — DRY, más fácil de testear y mantener

## Seguridad — Hallazgos Positivos

Verificaciones que resultaron SAFE en esta segunda pasada:

- **Prompt injection**: LLM prompts wrappean el diff en code fences. PR title/body/commits NO se inyectan en el system prompt. ✅
- **Race conditions**: Inngest serializa reviews por PR. SQLite WAL previene duplicados. OAuth codes son single-use. ✅
- **Cryptografía**: AES-256-GCM con IV de 12 bytes, tag de autenticación, `timingSafeEqual()` en todas las comparaciones HMAC. ✅
- **ncc bundle**: Sin secretos embebidos en el bundle de la action (4.3MB verificado). ✅
