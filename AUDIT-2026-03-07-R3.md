## Hallazgos y Propuestas de Mejora

### ALTO

#### 1. Circuit breaker definido pero NO usado

**Archivos**: `apps/server/src/lib/circuit-breaker.ts`, `apps/server/src/index.ts` línea 20
**Problema**: El `githubCircuitBreaker` está importado y expuesto en `/health/detailed`, pero `.execute()` no se usa en NINGUNA llamada real a GitHub API. Las llamadas a `fetchPRDiff()`, `getInstallationToken()`, etc. van directo sin pasar por el breaker.

Está implementado pero no conectado. Es como instalar un disyuntor eléctrico y no conectarle cables.

**Cambio propuesto**: Wrappear las llamadas a GitHub API:

```typescript
// En github/client.ts
export async function fetchPRDiff(token: string, repo: string, prNumber: number) {
  return githubCircuitBreaker.execute(async () => {
    const response = await fetch(`https://api.github.com/repos/${repo}/pulls/${prNumber}`, {
      headers: { Authorization: `Bearer ${token}`, Accept: 'application/vnd.github.v3.diff' },
    });
    if (!response.ok) throw new Error(`GitHub API error: ${response.status}`);
    return response.text();
  });
}
```

**Esfuerzo**: 1 hora
**Impacto**: Alto — el circuit breaker no sirve de nada si no se usa

#### 2. Error messages del runner exponen detalles de GitHub API

**Archivo**: `apps/server/src/github/runner.ts` ~línea 475
**Problema**: El error incluye el body raw de GitHub API:

```typescript
throw new RunnerCreationError('github_error', `GitHub API error: ${status} — ${body}`);
```

Y en `apps/server/src/routes/api/runner.ts` líneas 89, 96, 98, 120, el `err.message` se devuelve al cliente directamente. Un atacante puede obtener detalles internos de la API de GitHub.

**Cambio propuesto**: Loguear el detalle, devolver mensaje genérico:

```typescript
// En runner.ts (github/)
logger.error({ status, body, repo }, 'GitHub API error during runner creation');
throw new RunnerCreationError('github_error', 'Failed to communicate with GitHub API');

// En runner.ts (routes/api/)
return c.json({ error: err.code, message: 'Runner operation failed' }, 502);
```

**Esfuerzo**: 30 minutos
**Impacto**: Alto — previene information disclosure

#### 3. Rate limiter vulnerable a IP spoofing via X-Forwarded-For

**Archivo**: `apps/server/src/index.ts` líneas 117, 124, 131
**Problema**:

```typescript
keyGenerator: (c) => c.req.header('x-forwarded-for') ?? c.req.header('x-real-ip') ?? 'unknown',
```

`X-Forwarded-For` es un header controlado por el cliente. Un atacante puede enviar un valor diferente en cada request para bypassear el rate limit completamente.

**Cambio propuesto**: Si el server corre detrás de Render (que es el caso), confiar solo en el ÚLTIMO IP del header (el que Render agrega):

```typescript
function getClientIp(c: Context): string {
  const xff = c.req.header('x-forwarded-for');
  if (xff) {
    // Render/proxy adds the real client IP as the rightmost value
    const ips = xff.split(',').map(ip => ip.trim());
    return ips[ips.length - 1] || 'unknown';
  }
  return c.req.header('x-real-ip') ?? 'unknown';
}
```

**Esfuerzo**: 30 minutos
**Impacto**: Alto — el rate limiting actual es bypasseable

#### 5. Biome warnings en archivos de test (~80 warnings)

**Archivos**: Principalmente en `apps/action/src/tools/__tests__/*.test.ts` y `packages/db/src/queries.ts`
**Problema**: Non-null assertions (`!`) en ~25 lugares. No es un bug, pero ensucian el output de lint.

**Cambio propuesto**: Reemplazar `result[0]!` con `result.at(0)` + guard, o agregar `// biome-ignore` con justificación.

**Esfuerzo**: 1 hora
**Impacto**: Bajo — limpieza de lint

---

#### 6. Token cache: revocación tiene ventana de 5 minutos

**Archivo**: `apps/server/src/middleware/auth.ts` líneas 50-69
**Problema**: Si un token de GitHub se revoca, el usuario sigue autenticado hasta 5 minutos por el cache. Aceptable para la mayoría de casos, pero vale documentarlo.

**Cambio propuesto**: Reducir TTL a 2-3 minutos, o documentar el comportamiento como decisión de diseño.

**Esfuerzo**: 15 minutos
**Impacto**: Bajo — tradeoff aceptable entre performance y seguridad

---

#### 7. tsconfig de packages/types no excluye test files

**Archivo**: `packages/types/tsconfig.json`
**Problema**: No tiene `"**/*.test.ts"` en excludes, a diferencia de core y db. Si se agregan tests al paquete, se compilarían al dist.

**Cambio propuesto**: Agregar `"**/*.test.ts"` al exclude.

**Esfuerzo**: 2 minutos
**Impacto**: Bajo — consistencia

---

#### 8. Error handling inconsistente entre archivos de API

**Archivos**: `apps/server/src/routes/api/*.ts`
**Problema**: Los mensajes de error varían en formato:
- reviews.ts: `"Failed to fetch reviews"` (genérico)
- memory.ts: `"Failed to purge all memory observations"` (específico)
- repositories.ts: Sin error handling explícito (12 líneas)
- runner.ts: Devuelve HTTP 502, otros devuelven 500

**Cambio propuesto**: Estandarizar formato de error:

```typescript
{
  "error": "OPERATION_FAILED",
  "message": "User-facing description"
}
```

**Esfuerzo**: 1 hora
**Impacto**: Bajo-Medio — consistencia de API, facilita error handling en dashboard
